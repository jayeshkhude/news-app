Recommended stack: Neon + PostgreSQL

Neon is the best fit for NewsLens right now.

NewsLens has a clearly relational shape: articles connect to summaries, summaries connect to votes and comments, users connect to actions, and backups need durable storage. Neon gives us managed Postgres with the lowest operational overhead for a project that grows in bursts through ingestion runs and reader traffic spikes.

  • It keeps the database fully Postgres, so the app stays portable and standard.
  • Autoscaling matches the bursty load pattern of scheduled RSS ingestion plus reader traffic.
  • Branching is especially useful when we need to test schema changes without touching production data.
  • It already speaks the connection-string pattern this Flask app wants through DATABASE_URL.
Relational data Scheduled writes Low ops burden Future analytics ready

Why move off SQLite now.

SQLite is great for local development, but NewsLens is already beyond the sweet spot for a single file database. We have periodic ingestion, many tables, admin exports, votes, comments, and backups all touching the same store. That is exactly where a managed Postgres setup becomes the safer long-term choice.

  • Concurrent reads and writes are more reliable under real traffic.
  • Backups, restores, and observability are handled at the provider layer.
  • We can grow into reporting, search, and heavier analytics without changing database families again.

How our data fits Postgres.

This project already has multiple related entities. Postgres gives us the right foundation without giving up the simple table model we already use.

Content layer

Articles, summaries, headlines, source links, thumbnails, and publication timestamps.

User layer

Google-auth users, votes, comments, rate limits, and session-related data.

Ops layer

Backups, import scripts, exports, and future pipeline run logs.

Growth layer

Analytics, search indexes, category drilldowns, and possible vector features later.

DATABASE_URL = os.getenv("DATABASE_URL")

# Example

The backend now supports PostgreSQL through DATABASE_URL and keeps SQLite as a local fallback, so we can move production cleanly without breaking day-to-day development.

Migration checklist.

This is the cleanest path from the current SQLite setup to the new hosted Postgres stack.

01
Create the hosted Postgres database.

Spin up a Neon Postgres project, create the production database, and copy the provider connection string.

02
Set DATABASE_URL.

Add the Postgres connection string to the server environment so Flask and the scheduler both point at the same hosted database.

03
Run the migration script.

Use python backend/migrate_sqlite_to_postgres.py to copy current SQLite data into Postgres before cutover.

04
Verify writes from the pipeline.

Trigger one ingest run, check article and summary counts, then verify votes, comments, and admin export still work.

Recommendation

Primary production choice: Neon-hosted PostgreSQL.

Keep SQLite only as a local fallback, not as the deployed source of truth.
